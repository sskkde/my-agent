# API 性能基线

本文档记录 API 延迟的 CI 基线数据、阈值设定依据，以及如何运行和解读性能测试。

## CI 环境基线数据

性能冒烟测试在 CI 环境中运行，使用 `:memory:` SQLite 数据库和 `createApiServer` 搭建的 Fastify 实例。每个端点连续请求 20 次，取 p95（第 95 百分位）作为衡量指标。

### 端点阈值

| 端点 | 方法 | CI p95 阈值 |
|------|------|-------------|
| `/api/v1/health` | GET | < 200ms |
| `/api/v1/sessions` | GET | < 1000ms |
| `/api/v1/tools` | GET | < 200ms |
| `/api/v1/sessions` | POST | < 500ms |

### 阈值为什么偏宽松

CI 环境的资源不稳定，共享 runner 的 CPU、内存和磁盘 I/O 波动较大。这些阈值是刻意放宽的，目的是避免因环境抖动导致误报。它们不代表生产环境的实际性能水平，只作为回归检测的最低安全网。

具体来说：

- **health 和 tools** 属于轻量端点，不涉及数据库写入，CI 中通常在个位数毫秒完成，200ms 的阈值留出了近 20 倍的余量。
- **GET sessions** 需要查询数据库并序列化结果列表，1000ms 的阈值容纳了 CI 中偶发的磁盘抖动。
- **POST sessions** 涉及数据库写入，500ms 的阈值比 GET 端点稍宽，为写操作留出空间。

如果 CI 中某端点的 p95 接近阈值上限，说明可能存在性能退化，即使测试仍能通过。

## 运行性能测试

```bash
npx vitest run tests/performance/
```

测试会自动启动一个临时的 Fastify 服务器（随机端口），使用内存数据库，运行完毕后关闭。无需额外配置。

### 超时配置

- **用例超时**: 15 秒（`vitest.config.ts` 中的 `testTimeout`）
- **钩子超时**: 30 秒（`hookTimeout`，用于 `beforeAll`/`afterAll` 启动和关闭服务器）

### 查看详细延迟数据

测试运行时会在控制台打印每次请求的延迟值：

```
GET /api/v1/health - p95: 3.21ms
  All latencies: 1.05, 1.12, 1.08, ..., 3.21ms
```

如果需要更详细的输出，可以增加 `--reporter=verbose` 参数。

## 解读测试结果

### p95 的计算方式

测试收集 20 次请求的延迟，排序后取第 19 个值（索引 18）作为 p95。这意味着 20 次请求中最多允许 1 次超过 p95 值。公式：

```
p95Index = ceil(0.95 * 20) - 1 = 18
```

### 什么时候需要关注

| 情况 | 说明 |
|------|------|
| p95 超过阈值 | 测试失败，存在明确的性能退化 |
| p95 接近阈值 80% 以上 | 虽然通过，但余量不足，值得排查 |
| 单次延迟异常高 | 可能是冷启动或 GC 暂停，偶发可忽略，频繁则需调查 |
| 连续多次运行 p95 波动大 | CI 环境不稳定，考虑在本地复现确认 |

### 常见误报原因

- CI runner 资源争抢导致延迟飙升
- Node.js 冷启动（首次请求通常比后续请求慢）
- 并发测试运行时的端口冲突或资源竞争

## 生产环境推荐阈值

生产环境资源稳定，且用户对延迟更敏感。以下是建议的生产环境阈值，约为 CI 阈值的 1/5 到 1/4：

| 端点 | 方法 | 生产 p95 推荐 | 生产 p99 推荐 |
|------|------|---------------|---------------|
| `/api/v1/health` | GET | < 50ms | < 100ms |
| `/api/v1/sessions` | GET | < 200ms | < 500ms |
| `/api/v1/tools` | GET | < 50ms | < 100ms |
| `/api/v1/sessions` | POST | < 100ms | < 250ms |

这些值基于 CI 测试中观察到的典型延迟（远低于 CI 阈值），预留了合理的安全余量。实际生产阈值应根据真实监控数据调整。

### 监控建议

生产环境建议配合 Prometheus 指标（`agent_platform_request_duration_seconds`）持续观测，并设置告警规则：

- **P95 超过推荐阈值**: 触发 warning 告警
- **P99 超过推荐阈值的两倍**: 触发 critical 告警
- **延迟持续上升趋势**: 触发 warning 告警，即使尚未超过绝对阈值

告警规则可通过平台告警 API 管理，详见 `docs/observability/alerting-runbook.md`。
